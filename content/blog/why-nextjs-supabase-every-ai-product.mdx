Every product I ship starts with roughly the same decision: Next.js 15 on Vercel, Supabase for Postgres, Auth, and Row-Level Security. Not because it's trendy — because it removes an entire category of decisions that would otherwise slow down a solo build directed through AI.

## The core reason: RLS replaces a permissions layer

NexGuild has six distinct admin roles plus regular contributors. Building that access control as application-layer middleware means writing (and re-checking) permission logic in every single API route. Supabase's Row-Level Security pushes that enforcement into the database itself: a policy on the table decides who can read or write a row, full stop, regardless of which route touches it.

That matters enormously when you're building solo and fast. Every new feature inherits the same security guarantees automatically, because the guarantee lives in Postgres, not in a file that has to be remembered and re-applied.

## Why the App Router specifically

Server Components let me fetch data — reading `data/*.json` config files, querying Supabase, checking a user's session — without shipping that logic to the client at all. For a platform like NexGuild with real financial data (NexCoin balances, commission ledgers), that's not a performance nicety, it's a security boundary. The admin dashboard's sensitive queries never touch client-side JavaScript in the first place.

Route groups are the other underrated feature. NexGuild splits contributor-facing pages, organisation-facing pages, auth flows, the dashboard, and the admin panel into separate groups — all sharing one codebase, one deploy, one domain, with completely different layouts and auth requirements per group.

## What breaks without Postgres and real auth

xtoolkit.live is the counterexample that proves the point: it's a React SPA with an Express API and no database, because 44 free tools genuinely don't need persistent user data. That's the right call *for that product*. The moment a platform needs accounts, balances, or moderation — the NexGuild and eventual client-project pattern — Supabase's combination of Postgres, Auth, and RLS stops being a "nice to have" and becomes the only way to keep the system honest as it scales past routes you can manually audit.

Pick the stack based on what the product actually needs to guarantee, not on what's fastest to prototype. The five minutes you save skipping real auth and RLS on a real product cost a lot more than five minutes later.
